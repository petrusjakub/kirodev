# Interactive INDEX/MATCH lookup interface for premium Excel

The script was redesigned from a flat dump of all premium values (one section per age per gender) into an interactive lookup tool. The user now selects Gender and Age in two input cells at the top of each sheet, and INDEX/MATCH formulas instantly display the corresponding premium across all plans. Six backing data tables (four for Konvensional, six for Syariah) store 5000+ numeric values that the formulas reference. The `XlsxWriter` gained support for formula cells, numeric cells, data validation dropdowns, and merged cells.

**Watch for:** No IFERROR wrapper on formulas — invalid or out-of-range inputs display raw `#N/A` (confirmed). No data validation on the Age cell — users can enter values outside 0-85 (confirmed). Dead `start_row` parameter in `build_data_table` is misleading but harmless (confirmed).

## High-level view

The old approach generated hundreds of rows per sheet (one block per age-gender combo), making the file enormous and hard to navigate. The new approach puts a compact "dashboard" at the top (rows 1-17 for Konvensional, 1-23 for Syariah) with formulas that pull from data tables further down. The data tables use a composite key (`Gender_Age`) as the MATCH target, which avoids two-dimensional lookup complexity.

The `XlsxWriter` class grew a type system for cells — a 3-tuple `(value, style_id, cell_type)` where `cell_type` controls whether the cell is emitted as a shared string, raw number, or formula. This is the minimal extension needed to support INDEX/MATCH without pulling in a library dependency.

The main gap is input validation: the Gender dropdown constrains to "Pria"/"Wanita", but the Age cell accepts any value. Ages outside 0-85, non-integers, or blank cells all produce `#N/A` in every formula cell simultaneously, which may confuse non-technical users.

## Lookup architecture via composite keys

The design concatenates `$B$4&"_"&$B$5` (e.g., `"Pria_25"`) and matches it against a key column with entries like `Pria_0` through `Wanita_85`. Each benefit type gets its own 173-row block (1 header + 172 data rows = 86 ages x 2 genders). The formula pattern is:

```
INDEX(B$21:B$192, MATCH($B$4&"_"&$B$5, $A$21:$A$192, 0))
```

Each plan column gets its own formula pointing at the corresponding column in the data table. Konvensional has 30 formulas (8+7+8+7), Syariah has 39 (8+5+8+5+8+5).

None of the formulas are wrapped in IFERROR. If B5 is blank or contains a value > 85, every formula cell shows `#N/A`. For a tool meant to be handed to insurance agents, an IFERROR wrapper returning `""` or `0` would prevent a wall of errors on invalid input. The default values ("Pria" in B4, `0` in B5) mitigate first-open specifically, but subsequent edits to invalid values still expose the issue.

## Age input lacks constraint

Gender uses a list-type data validation (`"Pria,Wanita"`), so the user can only pick valid options. Age (B5) has no equivalent constraint. A `wholeNumber` data validation with `min=0, max=85` would prevent out-of-range entries, but the current data validation infrastructure only supports `type="list"`:

```python
data_validations = [("B4", '"Pria,Wanita"')]
```

Adding `type="whole"` would require extending `_build_sheet` to accept validation type as part of the tuple, or switching to a dict-based validation config.

## Dead parameter and redundant transform

`build_data_table(ages_data, benefit_key, plan_count, start_row)` accepts `start_row` but never references it. Positioning is handled implicitly by append order in `build_lookup_sheet`. The parameter creates a false expectation that the function cares about its absolute row position.

Similarly, the gender key derivation:

```python
gender_key = gender.lower().replace("pria", "pria").replace("wanita", "wanita")
```

is functionally equivalent to `gender.lower()`. The two `.replace()` calls map each string to itself — likely a leftover from a refactoring where the keys differed.
