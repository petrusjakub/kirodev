# MDWA product calculator via raw XML xlsx generation

This change adds `generate_mdwa_excel.py`, a ~980-line Python script that builds a 7-sheet Excel workbook (`Kalkulator_MDWA.xlsx`) for Manulife Dynamic Wealth Assurance plans A, B, and C. The script uses only the standard library (zipfile + string concatenation) to produce valid OOXML, matching the existing `generate_excel.py` pattern in this repo. It also removes two unrelated Chinese-language files (`generate_lemon_premium.py` and the corresponding `保费费率表.xlsx`).

**Watch for:** Arithmetic errors in the Simulasi sheet benefit calculations (possible), shared-string deduplication fragility when strings appear in multiple plans (confirmed), and the absence of any automated check that the hardcoded product data matches the source PDFs (confirmed).

## High-level view

The script follows the same raw-XML-in-zipfile approach already established by `generate_excel.py`. A `SheetBuilder` helper class encapsulates row/cell/merge-cell construction. Shared strings are collected in a pre-pass through hardcoded lists, then referenced by index in the sheet XML — a design that couples string collection order tightly to sheet construction order, and silently falls back to inline strings when a registration is missed.

All product data (benefit percentages, payout schedules, commission rates, competitor comparisons) lives as inline Python literals. Any rate update requires editing the script and re-running it.

The Simulasi sheet pre-computes example payouts as static text strings rather than Excel formulas, so the spreadsheet isn't interactive — an agent can't change a premium amount and see results recalculate. The sheet title says "Simulasi" (simulation) but the content is closer to pre-canned illustrations.

## Shared-string registration and silent fallback

The `main()` function collects strings into `all_strings` via the `add_str` helper, which deduplicates by tracking a `seen` set. Every string referenced in any `build_*_sheet` function must have been registered before `ss_map` is constructed. If one is missed, `SheetBuilder._cell_xml` emits an inline string (`t="inlineStr"`) instead of a shared-string reference (`t="s"`). This won't crash, but produces cells with different internal types — some Excel features (find/replace, sorting) may behave inconsistently across those cells.

Strings like `'+45%'`, `'+22.5%'`, and `'Masa Pembayaran'` appear in multiple `*_strs` arrays and are harmlessly deduplicated. The risk is the inverse case: a string used in sheet-building code but absent from all `*_strs` lists. No assertion guards against this.

## Simulation data accuracy

The Simulasi sheet hardcodes calculations like:

- Plan A Single 20yr Banding 2: `(9.5% + 6%) * 500M = 77.5M/yr * 15 yrs = 1,162.5M`
- Plan C 5yr 30yr Banding 2: `Tahapan 20% * 100M * 21 yrs = 420M`, Maturity `(510% + 45%) = 555%`

These combine base-rate percentages from Plan sheets with Banding 2 extras. The payout-year counts derive from payment-start rules on Plan A's sheet ("Akhir tahun polis ke-10" for PPP 5, "ke-15" for PPP 10, "ke-6" for Single). Manual cross-check of all three simulation scenarios against the rate tables shows consistency, but since the values are frozen text strings, any future rate adjustment that touches a plan sheet but not the Simulasi sheet will silently produce contradictory information within the same workbook.
