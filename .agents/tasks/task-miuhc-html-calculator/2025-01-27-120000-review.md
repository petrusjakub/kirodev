# HTML premium calculator generated from PDF extraction

A new `generate_html.py` script reuses the PDF extraction functions from `generate_excel.py` to produce a self-contained 61KB HTML file (`Kalkulator_Premi_MiUHC.html`) that serves as an interactive premium calculator for MiUHC insurance plans. The script serializes all premium data as JSON directly into the HTML, implements tab switching between Konvensional and Syariah products, and renders plan cards with real-time premium computation in the browser. No external dependencies ã no CDN, no fonts, no images beyond an inline SVG.

**Watch for:** Animation re-trigger after recalculation never fires correctly because `animationDelay` is set after cards already started their animation (confirmed). The `innerHTML` card-building concatenates plan metadata strings directly without escaping, which is safe only because the data is hardcoded ã but the pattern would become an XSS vector if plan names were ever user-sourced (possible). The "Semua Plan" section duplicates the same plans already shown in "Most Wanted" and "HEMAT 40%", creating visual redundancy rather than showing only the remaining plans (confirmed).

## High-level view

The generator uses template-replacement: Python extracts numeric data from two PDFs (reusing existing functions), serializes it as minified JSON, and injects it into an HTML template via placeholder strings (`__DATA_KONV__`, etc.) ã avoiding f-string conflicts with CSS/JS braces. The HTML output is self-contained at ~61KB.

The JavaScript rebuilds all cards via `innerHTML` on each dropdown change, which means animation state is lost. Cards start with `opacity:0` and rely on a CSS animation to fade in, but the stagger delay is set after the animation has already begun, producing a flash on recalculation.

The `buildCard` function concatenates plan metadata strings directly into HTML without escaping. All values originate from hardcoded Python constants so there's no current XSS surface, but the pattern is one data-source change away from a vulnerability.

## Animation lifecycle on recalculation

Cards are styled with `opacity:0` and `animation: fadeInUp 0.4s ease forwards` in the stylesheet. When `calculate()` runs, it replaces `innerHTML` of the results container, then queries all `.plan-card` elements and sets `animationDelay`. The browser has already parsed the new elements and started their animation (with the default 0s delay) by the time the loop runs. The stagger effect either doesn't appear or produces a flash where cards are invisible for a frame.

On subsequent recalculations (tab switch, dropdown change), every card briefly appears invisible (`opacity:0`) before the animation kicks in. A forced reflow (`void el.offsetHeight`) before setting delay, or removing the `opacity:0` default and using a JS-triggered class instead, would fix this.

## Plan duplication in the "Semua Plan" expandable

The "Most Wanted" section shows Jade ID, Jade, Topaz ID, Topaz. The "HEMAT 40%" section shows Jade Smart, Topaz Smart. The expandable "Semua Plan" section then renders all 13 plans again by iterating indices 0-12:

```javascript
for (var k = 0; k < 13; k++) {
    var name = PLAN_NAMES[k];
    html += buildCard(name, row[k], isSmart);
}
```

Six of the 13 cards appear twice on the page when the section is expanded. The intent appears to be showing the "other" plans (Diamond, Ruby, Emerald, Sapphire, Diamond Smart, Ruby Smart, Emerald Smart) ã but the implementation shows all plans including duplicates.

## innerHTML card concatenation without escaping

`buildCard` inserts values directly:

```javascript
'<div class="card-header">' + planName + '</div>'
'<span class="detail-val">' + wilayah + '</span>'
```

All values come from `PLAN_DETAILS`, a JSON object embedded at generation time from Python constants. The risk is future maintenance: if someone adds a plan name with `<` or `&` characters, or if the data source changes to include user-editable fields, this becomes an injection point. The fix is trivial (a 3-line escape function) and costs nothing.

## Generated artifact in source control

The 61KB HTML file contains the full premium dataset serialized as JSON. It's a build artifact ã running `python3 generate_html.py` reproduces it exactly. Committing it means any change to `generate_html.py` or the PDFs requires remembering to regenerate and re-commit the HTML, the HTML can drift from the generator if someone edits it directly, and PR diffs include ~600 lines of minified JSON data that obscure the actual code changes.

Whether this is intentional (for easy distribution without running Python) is a valid trade-off, but it should be a conscious decision.
